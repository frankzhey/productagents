# Acceptance Criteria 写作规范（PM agents 通用）

本规范是 product-planner / story-splitter / eng-reviewer 三个 agent 共享的 AC 单一来源。任何写 AC 或评审 AC 的环节都必须严格遵守。

**编号体系说明：**
- `A 类`：操作流程类约定（A-1 ~ A-8）
- `B 类`：字段 / 状态 / 校验类约定（B-1 ~ B-6）
- `C 类`：业务约定与扩展规则（C-1 ~ C-7）
- `①②③④⑤⑥⑦`：仅作为同一章节内的可读性序号，不用于跨文件引用

跨章节、跨文件引用一律使用 A/B/C 编号。

---

## §1 AC 格式强制规范（最高优先级 · 所有 AC 必须遵守）

每条 AC 必须采用 **GIVEN / WHEN / THEN 多行结构**，禁止压缩为一行、禁止使用箭头或斜杠串联。

### 1.1 ✅ 正确格式（多行 + 大写关键字 + 独立 AC 编号）

```
AC1：必填字段缺失时阻止提交
GIVEN 用户在订单创建页面
AND 已选择支付方式为"信用卡"
WHEN 用户点击"提交订单"按钮
AND 必填字段 `cardholderName` 未填写
THEN 系统在 `cardholderName` 字段下方显示报错："The Cardholder Name field is required."
AND 订单不提交
AND 焦点定位到 `cardholderName` 输入框
```

### 1.2 ❌ 禁止格式（连写 / 箭头 / 斜杠分隔 / 关键字混用大小写）

```
WHEN [字段A] 或 [字段B] 任一未填写时 → 按钮置灰
GIVEN 用户在表单/页面 WHEN 条件 THEN 操作
when 用户提交 then 显示报错
```

### 1.3 强制规则（每条都为阻塞性，违反必须重写）

1. 关键字 `GIVEN` / `WHEN` / `THEN` / `AND` / `BUT` 必须**全大写**
2. 每个关键字**独占一行**，行尾不接箭头（→）、斜杠（/）或其他分隔符
3. 同一前提下的多个条件用 `AND` **换行追加**，不允许"WHEN A 或 B"压缩为一行
4. 一条 AC 内只能出现 **一组** GIVEN-WHEN-THEN（可有多个 AND）
5. 多场景必须**拆为多条独立 AC**，编号 AC1 / AC2 / AC3，不允许在一条 AC 内塞多场景
6. **字段名必须用反引号** \`fieldName\` 标记，避免与中文描述混淆
7. AC 中**禁止出现 UI 视觉描述**（按钮颜色、字号、Tab 数量、布局），这些由 UX Prototyper 决定
8. 每条 AC 必须有标题（如 `AC1：必填字段缺失时阻止提交`），方便评审引用

## §2 AC 覆盖规范（按 Story 类型分类，强制执行）

在写 AC 之前，先判断 Story 类型并对号入座以下规范。

### 2.1 操作流程类 Story（用户完成步骤操作，有触发→反馈路径）— 必须覆盖 5 类

| # | 类别 | 编号 | 强制要求 |
|---|------|:---:|---------|
| ① | 操作触发范围 | **A-1** | 对哪类用户 / 数据执行；范围是什么（考点 / Sitting / 产品 / 全量）；哪些情况例外 |
| ② | 权限与配置依赖 | **A-2** | 权限前置条件写进 GIVEN；多角色分开写 AC；覆盖"有权限"和"无权限"两种场景 |
| ③ | 弹窗所有关闭方式 | **A-8** | Confirm / Cancel / 右上角 × 各自独立 AC；副作用（数据是否保留 / 操作是否执行）明确说明 |
| ④ | 幂等性 / 重复触发 | **B-4** | 显式说明：允许重复触发（重复无副作用）or 不允许重复触发（触发限制和条件）。禁止不写。 |
| ⑤ | 系统异常处理 | **B-6** | 无第三方：通用后端报错 + 操作中断说明 + 用户重试路径；有第三方：区分系统报错 / 业务规则不满足 / 并发限制 3 类，各自独立 AC |

### 2.2 列表查询类 Story（以筛选 + 列表为核心展示数据）— 必须覆盖 7 类

| # | 类别 | 编号 | 强制要求 |
|---|------|:---:|---------|
| ① | 页面访问权限 | **A-2** | 第一条 AC；哪些角色可访问；无权限角色的拒绝行为 |
| ② | 页面默认加载状态 | **A-3** | 明确三类之一：「默认空需主动 Search」/「默认全量」/「默认近 N 天 / 按条件」；默认排序字段 + 方向 |
| ③ | 筛选项规则 | **A-4** | 每个筛选项用表格列出：名称 / 类型 / 是否必填 / 搜索方式 / 特殊规则；条件间 AND/OR 关系；联动关系（或"独立无联动"） |
| ④ | 字段展示规范 | **B-1** | 合并表格：字段名 / 取值规则 / 展示格式 / 特殊规则（颜色 / 粗细 / N/A 条件）；状态映射关系独立用映射表，不内嵌文字 |
| ⑤ | 排序与分页 | **A-5** | 默认排序字段 + 方向；默认每页条数；筛选变更后分页重置到第一页 |
| ⑥ | 空状态 | **A-6** | 区分 4 类：硬空（拦截查询 + 提示文案）/ 软空（"-" 占位）/ 查询无结果（No data）/ 状态性空（带引导文案） |
| ⑦ | Reset / Export / Detail 操作 | **A-7** | Reset：清空 + 恢复默认排序 + 分页重置；Export：必选项校验 + 全量导出（非仅当前页）+ 文件命名 + 上限 + 空数据行为；Detail：跳转页面 or 弹窗 |

> 一个 Story 可能同时是两种类型（如列表页带操作按钮），此时两套覆盖规则都要满足。

## §3 AC 额外强制规则（所有 Story 类型）

### 3.1 状态机完整覆盖（C-1）

凡 AC 涉及状态字段（Status / State），必须：
- 列出完整状态值列表（包括 N/A 等特殊值及触发条件）
- 每种状态对应的展示规则用表格定义（状态值 / 展示文案 / 颜色加粗 / 可操作行为）
- 状态转移的触发条件明确说明（什么操作触发什么状态变化）

禁止：用"状态按系统逻辑变化"代替具体状态值列举。

### 3.2 按钮置灰规则（B-3）

凡 Story 包含提交 / 确认按钮，必须按以下多行格式各自独立 AC（禁止用 → 或 / 压缩成一行）：

```
AC：必填字段未完成时按钮不可点击
GIVEN 用户在 [页面名称]
AND 必填字段 `fieldA` 未填写
WHEN 用户查看 [按钮名称] 按钮
THEN 按钮处于不可点击状态（disabled）
AND 鼠标悬停时显示提示："请完成所有必填字段"

AC：所有必填字段完成后按钮可点击
GIVEN 用户在 [页面名称]
AND 所有必填字段（`fieldA`、`fieldB`、`fieldC`）均已填写并通过格式校验
WHEN 用户查看 [按钮名称] 按钮
THEN 按钮处于可点击状态（enabled）
```

若按钮在不同状态下文案不同，每种状态拆为独立 AC，禁止合并。

### 3.3 表单校验完整性（B-2）

表单提交 AC 必须覆盖 5 个要素：

1. **触发时机**（输入时实时校验 / 点击提交时）
2. **校验类型**（必填 / 格式 / 唯一性 / 业务规则）
3. **展示位置**（字段下方 / 右上角 Toast / 弹窗）
4. **错误文案**（精确到字符级，中英文各自注明）
5. **校验后数据状态**（输入框是否清空 / 弹窗是否关闭 / 用户如何重试）

## §4 AC 写法模板（按场景对号入座）

### 4.1 操作触发范围模板（A-1）

```
AC：操作触发范围
GIVEN 用户具备 [角色] 权限
AND 数据状态为 [状态值]
WHEN 用户执行 [操作]
THEN 系统对 [范围：考生 / 记录 / 数据] 执行 [动作]
AND 当 [例外条件1] 时，[例外行为]
AND 当 [例外条件2] 时，[例外行为]
```

### 4.2 幂等性模板（B-4，二选一必写）

**方式 1：允许重复触发**

```
AC：允许重复触发
GIVEN 用户已成功执行过一次 [操作]
WHEN 用户再次执行 [操作]
THEN 系统再次执行 [操作内容]
AND [已完成 / 已存在的数据] 不受影响
```

**方式 2：不允许重复触发（有限制）**

```
AC：不允许重复触发
GIVEN 用户在 [时间范围 / 条件] 内已执行过 [操作]
WHEN 用户再次尝试执行
THEN 系统 [阻止 / 合并 / 累积]，不创建新记录
AND 提示用户：[提示文案]
```

### 4.3 第三方错误模板（B-6，必须区分 3 类）

**类型 1：第三方系统报错（接口超时 / 500）**

```
AC：第三方系统报错
GIVEN 用户已触发 [需调用第三方的操作]
WHEN [第三方系统名] 返回错误（超时 / 500）
THEN 展示错误文案："[文案]"
AND [按钮文案变化]
AND 用户可重试（重试机制：[次数 / 退避策略]）
```

**类型 2：业务规则不满足（比对失败 / 阈值不达标）**

```
AC：业务规则不满足
GIVEN 用户已触发 [操作]
WHEN [第三方系统名] 返回 [具体结果值]
THEN 系统判定 [Fail / 拒绝 / 警告]
AND 展示文案："[文案]"
AND [具体结果值] → [展示文案] 的映射关系参见下表

| 结果值 | 展示文案 | 标记 |
|-------|---------|------|
| ...   | ...     | ...  |
```

**类型 3：并发 / 频次限制**

```
AC：并发频次限制
GIVEN 用户已触发 [操作]
WHEN 调用次数超过 [N] 次 / 时间窗口
THEN 系统 [降级 / 排队 / 拒绝]
AND 展示提示文案："[文案]"
```

### 4.4 筛选项规则模板（A-4）

```
AC：筛选项配置
GIVEN 用户进入 [页面名称]
WHEN 用户配置以下筛选项
THEN 系统按以下规则执行查询：

| 筛选项 | 类型 | 是否必填 | 搜索方式 | 特殊规则 |
|--------|------|---------|---------|---------|
| `fieldA` | 日期范围 | 必填 | — | 可选范围不超过 7 天 |
| `fieldB` | 下拉单选 | 可选 | — | 可选值：X / Y / Z |
| `fieldC` | 文本输入 | 可选 | 精确匹配 | — |

AND 多个筛选项同时配置时，条件之间为 AND 关系
AND [联动规则 / "各筛选项之间无联动关系，相互独立"]
```

### 4.5 字段展示模板（B-1）

```
AC：列表字段展示
GIVEN 用户在 [页面名称] 列表
WHEN 系统加载列表数据
THEN 列表按以下字段规范展示数据：

| 字段名 | 取值规则 | 展示格式 | 特殊规则 |
|-------|---------|---------|---------|
| `fieldA` | [数据来源] | DD/MM/YYYY | — |
| `fieldB` | 状态映射见下表 | [文案] | Fail 时标红加粗 |

AND 状态映射关系：

| 系统值 | 展示文案 | 标记 |
|-------|---------|------|
| `valueA` | "文案 A" | — |
| `valueB` | "文案 B" | 标红加粗 |
```

### 4.6 Export 操作模板（A-7）

```
AC：Export 必选项校验
GIVEN 用户在 [页面名称]
AND 必选筛选项 `fieldX` 未填写
WHEN 用户点击 Export 按钮
THEN 系统展示校验提示："[提示文案]"
AND 不执行导出

AC：Export 全量导出
GIVEN 用户在 [页面名称]
AND 必选筛选项均已填写
AND 当前查询结果有数据
WHEN 用户点击 Export 按钮
THEN 系统按当前筛选条件导出全量数据（非仅当前页）
AND 导出文件格式为 Excel
AND 文件命名规则：[文件名格式，如 `{业务名}_{YYYYMMDD_HHmmss}.xlsx`]
AND 单次导出上限 [N] 条

AC：Export 空数据行为
GIVEN 用户在 [页面名称]
AND 必选筛选项均已填写
AND 当前查询结果为空
WHEN 用户点击 Export 按钮
THEN 系统导出仅包含表头的 Excel 文件，无数据行
```

## §5 AC 自主补全分级（强制 · "先写后 review"）

为避免 PRD 中出现大量 Open Question 阻塞评审，遇到信息缺口时按以下三级处理。

### 5.1 🟢 直接补全（无需标注）— 通用交互模式，按下列默认值直接写入 AC

| 场景 | 默认补全规则 |
|------|------------|
| 必填字段校验文案 | `The [Field Name] field is required.`（英文）/ `[字段名]为必填项`（中文） |
| 级联筛选项重置 | 上游变更时下游自动清空（保留筛选 UI，清值为空） |
| 提交按钮置灰 | 必填项未填或未通过校验时不可点击（按 §3.2 多行格式） |
| 字段空值展示 | 默认展示 `—`（半角破折号），不显示 `null` 或空白 |
| 操作成功后列表刷新 | 列表自动刷新，保留当前筛选和排序，分页重置到第 1 页 |
| 不可逆操作二次确认 | Confirm / Cancel / × 三种关闭方式各自独立 AC，Cancel 和 × 行为一致：操作不执行，弹窗关闭 |

### 5.2 🟡 补全 + 标注 `[假设]`— 业务可推断但需 PM 确认

| 场景 | 补全策略 |
|------|--------|
| 二次确认弹窗的具体文案 | 写入合理默认文案 + `[假设]` 标记 |
| 异常报错的具体话术 | 写入通用文案（如"系统繁忙，请稍后重试"）+ `[假设]` 标记 |
| 无明确指示时的默认排序字段 | 选择业务最常用字段（如创建时间倒序）+ `[假设]` 标记 |
| Export 文件命名规则 | 使用 `{业务名}_{YYYYMMDD_HHmmss}.xlsx` + `[假设]` 标记 |

`[假设]` 标记示例：

```
THEN 系统弹出确认弹窗，文案为："确定取消该订单吗？取消后无法恢复。" [假设]
```

### 5.3 🔴 不可补全 · 必须 Open Question — 业务决策性强、补错代价高

以下场景禁止 agent 自行补全，必须列入 PRD 末尾的 `Open Questions` 区块：

- 角色权限矩阵（谁能看 / 谁能操作 / 数据范围隔离）
- 计算公式（财务、积分、评分、退款等强业务规则）
- 第三方接口的错误码 → 业务行为映射（依赖外部接口文档）
- MVP / Phase 2 范围切分
- 通知 / 邮件 / 推送的具体触发模板与收件人规则
- 数据同步的失败兜底策略（重试次数、降级路径、人工干预入口）

**补全分级原则**：宁可写错让 PM 改，也不留空让评审现场补；但业务决策性问题不能假装"自己懂"。

## §6 各 agent 使用方式

| Agent | 使用范围 | 调用时机 |
|-------|---------|---------|
| **product-planner** | §1 - §5 全部 | Step 6 写 Acceptance Criteria 时；Step 10.5 Quality Gate 自检时 |
| **story-splitter** | §1 - §5 全部 | 输出 Story List 之前；写每个 Story 的 AC 时 |
| **eng-reviewer** | §1 - §3 作为合规校验清单 | Section 17.0 AC 合规校验时；评审 PRD 中每个 Story 的 AC |

### 加载方式（Copilot 环境）

由于 Copilot 不支持 SKILL 自动 description 触发，所有 agent 必须在工作流早期**显式 Read 本文件**。

**正确路径（相对项目根目录）：**

```
Read skills/ac-writing-spec/SKILL.md
```

> ⚠️ 不要使用 `PM agents/skills/...` 这种带前缀的路径，本仓库实际目录结构是 `skills/` 直接挂在项目根下。

加载位置建议：
- product-planner：「在执行任何任务前」第 3 条「强制依赖加载」
- story-splitter：「执行前规则」第 4 条「强制依赖加载」
- eng-reviewer：「执行前规则」第 7 条「强制依赖加载」
