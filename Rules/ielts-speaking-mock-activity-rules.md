## Layer 1 — 业务规则

### 1.1 提交控制规则

- **一次性提交**：每个 WeChat `unionId` 在本次活动中**仅可提交一次录音**
  - 此限制由**后端强制执行**，不依赖前端控制
  - 后端需在两个节点执行 unionId 唯一性校验：① 获取上传 URL 阶段 ② 写入提交记录阶段
  - 数据库须对 `unionId` 字段设置唯一约束（unique constraint）
  - 重复提交时后端返回 **409 Conflict**（非 5xx），前端视 409 为"已提交"状态处理

- **前端锁定规则**：考生已提交录音后（`status = "submitted"` 或 `"scored"`），前端**不得展示录音入口**（「开始录音」按钮、操作面板均隐藏）
  - 前端通过刷新时重新查询后端状态实现，不依赖本地缓存

### 1.2 活动状态规则

- 活动状态（`activity_status`）分为两个值：`active`（进行中）/ `ended`（已结束）
- **活动结束状态仅影响"未提交"用户**的 CTA 展示：
  - 未提交用户：CTA 按钮展示为"谢谢参与，活动结束"，置灰不可点击，Banner 切换为结束版本
  - 已提交用户（`submitted` / `scored`）：不受影响，按原状态流程展示
- 活动状态切换由管理员通过后台手动操作，前端通过刷新感知（无需实时推送）

### 1.3 题目分配规则

- 38 套 Part 2 题目（图片形式）由**后端按 `unionId` 固定分配**，不因页面刷新或重新进入而重新随机
- 每个 `unionId` 在本次活动全程使用同一套题目，支持数据采集追溯
- 分配记录须与 `unionId` 关联存储于后端

### 1.4 状态展示保守策略

- 前端在用户状态接口**未返回结果期间（loading 中）**，默认隐藏所有操作入口（录音按钮、操作面板、CTA 等）
- 状态接口失败时（超时 / 5xx），前端展示"状态加载失败，请刷新重试"，**不允许在状态未确认时展示录音入口**

### 1.5 用户状态流转（不可逆）

| 状态值 | 含义 | 触发条件 |
|--------|------|---------|
| `unsubmitted` | 未提交 | 初始状态 |
| `submitted` | 已提交，待评分 | 后端写入提交记录成功 |
| `scored` | 评分完成 | 成绩 Callback 回传后更新 |

- 状态流转单向不可逆：`unsubmitted → submitted → scored`
- 数据库状态更新不允许回退

### 1.6 CTA 按钮状态机

| 用户状态 | 活动状态 | CTA 文案 | 可否点击 | 跳转目标 |
|---------|---------|---------|---------|---------|
| `unsubmitted` | `active` | "立即参与" / "开始口语测评" | ✅ 可点击 | 录音页 |
| `submitted` | `active` / `ended` | "等待测评结果" | ❌ 置灰 | 无 |
| `scored` | `active` / `ended` | "查看结果" | ✅ 可点击 | 成绩页 |
| `unsubmitted` | `ended` | "谢谢参与，活动结束" | ❌ 置灰 | 无 |

## Layer 2 — 工程模式

### 2.1 文件上传架构（预签名 URL 直传）

- 音频文件上传采用**预签名 URL 直传模式**（前端直传 OSS/COS，不经后端中转大文件）
- 流程：① 前端调用后端接口获取预签名 URL → ② 前端使用 `wx.uploadFile` 直传云存储 → ③ 上传成功后前端通知后端写入状态
- **URL 过期处理**：前端在上传前检测 URL 有效期，过期时自动无感刷新一次（重新调用获取 URL 接口），超出 1 次后提示用户手动重试
- 上传超时阈值：60 秒（超出视为失败，触发重试）

### 2.2 幂等设计约定

- **409 vs 5xx 区分原则**：后端幂等接口在 DB 写入冲突（`unionId` 重复）时返回 **409**，而非 5xx
- 前端对 409 的处理：做**业务判断**而非错误处理
  - 「获取上传 URL」接口返回 409 → 展示已提交状态，不进入上传流程
  - 「上传完成通知」接口返回 409 → 视同成功，直接跳转等待页
- 数据库须实现 `unionId` 唯一约束，并发请求时 DB 层保证只写入一条记录

### 2.3 成绩 Callback 安全规范

- 成绩 Callback 接口**必须实现来源验证**（HMAC 签名 / Bearer Token / IP 白名单，具体方案与 AI 评分服务对齐）
- 验证失败返回 **401/403**，记录安全告警日志（来源 IP + 请求内容摘要 + timestamp）
- **幂等处理**：同一 `unionId` 的 Callback 重复发送时，后端检测 `status = "scored"` 不重复更新，返回 200 通知 AI 服务停止重试
- 建议实现死信队列处理持续失败 Callback（最多 3 次重试，指数退避）

### 2.4 录音状态机设计（前端）

录音页状态单向流转（不可逆）：

```
就绪（Ready）
  ↓ 点击「开始录音」
录音中（Recording）—— 显示倒计时
  ↓ 手动点击「结束录音」或倒计时归零
录音完成（Recorded）—— 显示操作面板：「重新录音」「播放」「上传录音」
  ↓ 点击「上传录音」，上传成功
已提交（Submitted）—— 隐藏所有录音入口
```

- 各状态对应的 UI 入口**互斥展示**，不允许同时显示多个状态的操作入口
- 上传进行中（过渡态）：所有操作按钮（包括「重新录音」「播放」）同步置灰

### 2.5 微信 API 集成规范

| API | 用途 | 关键注意点 |
|-----|------|----------|
| `wx.login` + `code2session` | 登录获取 `unionId` | AppSecret 仅在服务端使用，不可暴露前端 |
| `wx.authorize (scope.record)` | 麦克风权限 | 拒绝后只能通过 `wx.openSetting` 引导，不可再次自动弹窗 |
| `wx.getRecorderManager` | 录音 | `format: 'mp3'` 兼容性需研发验证，备选方案：`aac` + 服务端转码 |
| `wx.uploadFile` | 文件直传 | 目标域名需配置在小程序后台「uploadFile 合法域名」白名单 |
| `InnerAudioContext` | 录音回放 | 使用本地 `tempFilePath`，不上传到服务端再回放 |
| `wx.shareAppMessage` | 好友分享 | 需在 Page 中配置 `onShareAppMessage`，分享卡片素材由后台配置 |
| `wx.shareTimeline` | 朋友圈分享 | 需小程序后台开通「分享到朋友圈」能力，需配置 `onShareTimeline` |
| `wx.navigateTo` / `wx.redirectTo` | 页面跳转 | 等待页跳转建议使用 `redirectTo`（防止用户返回录音页） |

### 2.6 Logging 规范

关键业务操作日志必须包含以下字段：

| 字段 | 说明 |
|------|------|
| `trace_id` | 全链路追踪 ID |
| `union_id` | 微信用户标识 |
| `service_name` | 服务名称 |
| `error_level` | 错误级别（INFO / WARN / ERROR） |
| `timestamp` | UTC 时间戳 |
| `error_type` | 错误类型（如适用） |

**必须记录日志的节点**：
- 微信授权登录
- 获取预签名 URL（成功 / 失败）
- 文件上传（成功 / 失败）
- 提交记录写入（成功 / 失败 / 幂等拦截）
- 成绩 Callback 接收（成功 / 验证失败 / 幂等拦截）
- 活动状态查询（管理员操作变更）

## Layer 3 — AC 约定

### 3.1 上传错误 AC 三类分类规范

所有包含上传操作的 Story，AC **必须明确以下三类错误**，各自独立 AC 场景：

| 类型 | 说明 | 典型响应 | 前端处理 |
|------|------|---------|---------|
| ① 系统报错 | 接口超时 / 服务 5xx | HTTP 5xx 或 timeout | 通用后端报错 + 重试按钮 |
| ② 业务规则不满足 | 文件格式错误 / 文件过大 / 已提交 | HTTP 4xx（具体错误码） | 精确错误文案 + 引导重录 |
| ③ 并发限制 | 频次超限 | HTTP 429 Too Many Requests | 冷却提示 + 按钮置灰倒计时 |

### 3.2 弹窗关闭方式覆盖规范

所有包含弹窗的 Story，AC **必须覆盖以下四种关闭方式**：

| 关闭方式 | 副作用说明 |
|---------|----------|
| 点击「确认」 | 执行目标操作（如删除录音、提交等），弹窗关闭 |
| 点击「取消」 | 不执行操作，数据保留，弹窗关闭，页面状态不变 |
| 点击右上角「×」 | 等同取消，数据保留，弹窗关闭 |
| 点击蒙层外区域 | 等同取消，数据保留，弹窗关闭 |

### 3.3 按钮置灰约定

- 上传进行中：所有操作按钮（「重新录音」「播放」「上传录音」）**同步置灰**，防止并发操作
- 上传完成或失败后，按钮状态依结果恢复（成功跳转 / 失败展示重试）
- 状态接口加载期间（loading 中）：所有 CTA 按钮隐藏或置灰，不允许在状态未确认时可交互

**提交按钮置灰规则模板**（适用于所有含提交/确认按钮的 Story）：
```
WHEN [必填字段A] 或 [必填字段B] 任一未满足时 → 按钮置灰，不可点击
WHEN 所有前置条件均满足 → 按钮高亮，可点击
WHEN 操作进行中（loading）→ 按钮置灰，防止重复触发
```

### 3.4 微信 API 场景覆盖规范

涉及微信 API 调用的 Story，AC **必须覆盖以下两类场景**：

| 类型 | 覆盖要求 |
|------|---------|
| API 调用失败（onError / 超时） | 展示具体错误提示 + 用户重试路径 |
| 权限被拒绝 | 按钮置灰 + 引导文案 + 「前往设置」快捷入口（`wx.openSetting`） |

### 3.5 状态机 AC 覆盖规范

凡 Story AC 涉及状态字段（`status` / `activity_status` / 录音状态），必须：
- 用**映射表**列出完整状态值（包括边界值）
- 每种状态对应的展示规则明确说明
- 状态转移的触发条件明确写出

**本项目用户状态完整映射**：

| status 值 | 主页 CTA 文案 | 录音页入口 | 等待页展示 | 成绩页入口 |
|-----------|-------------|----------|----------|----------|
| `unsubmitted` | "立即参与" / "开始口语测评"（高亮） | 展示 | 不展示 | 不展示 |
| `submitted` | "等待测评结果"（置灰） | 不展示 | 展示 | 不展示 |
| `scored` | "查看结果"（高亮） | 不展示 | 可跳转结果页 | 展示 |

### 3.6 幂等性 AC 覆盖规范

每个 Story 中涉及"重复触发同一操作"的场景，AC **必须明确说明**：
- 允许重复触发（重复无副作用）→ 说明幂等保证方式
- 不允许重复触发 → 说明拦截机制（前端 / 后端 / 数据库层）和返回行为
